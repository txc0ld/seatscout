## Philosophy

The atlas is designed to be:

- curated, not exhaustive
- deterministic, not runtime-generated by an external model
- high-level and useful for pre-purchase orientation
- easy to extend without rewriting UI logic

## Record Structure

Each raw venue record includes core metadata such as:

- name
- city
- country
- region
- continent
- coordinates
- type
- capacity
- primary use
- roof type
- opened year
- hero note
- general tips

Derived profile fields include:

- `slug`
- `sections`

Some newer records also include hidden `sourceRefs` for future provenance backfill.

## Section Generation

Section profiles are generated from venue layout archetypes:

- `oval`
- `rectangle`
- `arena`
- `amphitheater`
- `horseshoe`

For each venue, the generator creates:

- level bands
- zone distribution
- row counts
- ratings for view, comfort, value, access, and atmosphere
- pros and tradeoffs
- seating guidance such as best rows and rows to avoid

This keeps the atlas consistent across venues while still respecting venue type.

## Sourcing Standard

When tightening or adding metadata:

- prefer official venue or operator sources first
- use league or team venue pages when operator material is incomplete
- use venue PDFs or visitor guides when that is the best primary source available
- avoid unsourced crowd-edited facts when a primary source exists

If a field is approximate or operator sources conflict, keep the more conservative value and add a note in `sourceRefs` where possible.

## Update Workflow

1. Add or revise the raw venue record.
2. Keep naming and regional formatting consistent with neighboring entries.
3. Run `npm run build`.
4. Spot-check the affected venue in the UI.
5. Update README counts if coverage totals changed.
